# 循迹赛道第二题闭环行驶方案

## 需求 / 验收标准
- 小车在 A 点就位且灰度传感器初始处于白底；按下 Key1 后，按 A→B→C→D→A 的顺序完成一圈。
- A→B 与 C→D 为白底直线段：使用已跑通的直线保持能力行驶，灰度传感器重新检测到黑线即分别判定抵达 B、D。
- B→C 与 D→A 为黑色半圆弧：使用已跑通的灰度循迹能力行驶；灰度传感器失去黑线即分别判定抵达 C、A。
- 每次黑白状态切换均停车 1 s，依次发生在 B、C、D、A；到达 A 的 1 s 暂停结束后保持停车。
- 本轮不实现声光提示；停车 1 s 用作节点提示的临时替代。
- 首版沿用当前已跑通的速度；实物全程计时不大于 30 s。若超时，再进入提速调参轮次。

## 约束
- 赛道除两段黑色半圆弧外均为白色，无 A/B/C/D 额外定位标记。
- 直线段终点不得依赖编码器里程计判定，只以灰度传感器对黑线的重新识别判定。
- 节点必须采用状态机使能、连续采样确认和最小行驶时间保护，避免黑线边缘抖动造成重复停车或误切换。
- 黑线进入与离开均以连续 3 次 10 ms 灰度刷新结果一致为确认条件，首版确认延迟约 30 ms。
- APP 仅编排状态机；白底直行经现有直线保持 APP，黑线弧线经现有循迹 APP，不新增对 DriverLib/SysConfig 的直接依赖。

## MVP 边界
- 包含：Key1 启动、四段状态编排、四次 1 s 暂停、最终停车、全程计时验证。
- 不包含：声光硬件驱动、自动提速、基于里程计的点位定位、赛道异常后自动重定位。

## 已选实现方案
- 新建 `empty/User/APP/app.c` 作为赛题状态机实现，新增配套 `app.h` 供 CPU 调度器调用；`app_drive_mode.c` 在本题中不使用。
- 状态顺序：等待 Key1 → 白底直行至 B → B 暂停 → 弧线循迹至 C → C 暂停 → 白底直行至 D → D 暂停 → 弧线循迹至 A → A 暂停 → 完成停车。
- 白底状态调用 `app_straight_drive`；弧线状态调用 `app_line_follow`；暂停和完成状态均停止电机，使用 BSP 毫秒时基非阻塞计时。
- 每个行驶状态只接受与本段目标匹配的灰度边沿：直线只接受“无黑线→有黑线”，弧线只接受“有黑线→无黑线”。

## 实施清单（待用户授权后执行）
1. 新建 `app.[ch]`：实现赛道状态、3 次连续灰度确认、非阻塞 1 s 暂停及最终停车逻辑。
2. 在 `scheduler` 刷新 Key1 并调用新 `app`，移除当前直接调用循迹 APP 的临时入口；不使用 `app_drive_mode`。
3. 仅在状态机复用现有接口遇到必要缺口时，最小化修改 `app_line_follow` 或 `app_straight_drive`。
4. 构建并上板验证：Key1 启动、B/C/D/A 四次暂停的顺序与最终停车。
5. 以秒表实测全程；若超过 30 s，再单独进入速度参数调优轮次。
