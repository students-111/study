# 闭环控制符号陷阱与对照实验诊断法

> 本文件在遇到以下情形时按需加载：
> - 闭环系统（PID / 前馈 / MPC）出现"看起来能动但方向反、震荡、死循环放大"的症状
> - 改过增益 / 限幅 / 保护链多轮仍无改善
> - Kp / Kd / 限幅 调参像"拍脑袋盲盒"，没有物理依据
>
> 与 `systematic-debugging.md`（四阶段通用方法）互补：本文件专注**控制系统特有的符号陷阱**。

---

## 铁律

**先验证方向，再调参数。**

Kp / Kd / phi_max / rate_limit 都是**放大器或限幅器**，它们不能修复方向性错误，只能放大或压制。符号错了，**所有下游调参都在给错误方向加油**。

---

## 什么是"符号陷阱"

闭环控制中任何从测量到执行的通路（误差 → 控制量 → 执行器 → 物理响应）都隐含一个**符号假设**。当以下任一环节翻转却没同步翻转其他环节时，产生符号错误：

| 环节 | 符号来源 |
|------|---------|
| 传感器轴定义 | IMU yaw CCW 为正 or CW 为正？厂商不同约定 |
| 坐标系 | 车头 +X or +Y？path 起点切线方向？ |
| 误差极性 | error = target - measured 还是 measured - target？ |
| 控制律几何 | κ = +2sin(α)/Ld 还是 -2sin(α)/Ld？ |
| 前馈符号宏 | `LEAN_FEEDFORWARD_SIGN = +1` or `-1`？ |
| 执行器极性 | servo 正数是左打还是右打？motor 正数是前进还是后退？ |
| PID 增益符号 | Kp 正数对应"加大 error 时加大输出" or 相反？ |

一个系统有 7 处符号，**任何一处翻了其他没跟上就是 bug**。

## 为什么符号错最难发现

1. **编译不会报错**：浮点数乘 -1 语法合法
2. **系统"看起来能动"**：有输出、有响应、有波形
3. **测试覆盖不到**：单元测试测不出 α→φ→servo→car 完整物理闭环的方向正确性
4. **Kp 大的时候症状是"震荡"而非"静止"**：震荡很容易被误诊为"调参问题"
5. **历史遗留难追溯**：符号可能在某次重构时应该翻转但没翻转，后续所有 commit 都基于错误假设

## 典型症状清单

命中其中任意 3 条以上，立即怀疑符号错：

- [ ] 启动瞬间执行器方向与期望反
- [ ] 前馈关掉（置 0）后系统反而更稳
- [ ] PID Kp 越大震荡越厉害（正常系统应有一个临界点）
- [ ] Kp 降到很低才勉强稳住，但跟踪滞后严重
- [ ] 调 phi_max / rate_limit / 保护阈值都治标不治本
- [ ] 某些时刻 error 小但输出反向增大
- [ ] 相同代码在旧路径工作，重构路径后不工作（坐标变换导致符号应翻但没翻）

## 诊断核心方法：观察模式对照实验

**关键思想**：在不改变硬件和代码的前提下，**提供一条"只采样不控制"的路径**，和正常闭环并行对照。隔离出单一变量。

### 实施方法

需要能把"控制器 → 执行器" 这条写通道**运行时静音**但保留计算和遥测。做法：

1. 新增一个 observe/manual 标志（static uint8_t）
2. 在控制器写执行器的位置加守卫：`if(!observe_mode) { executor_write(cmd); }`
3. 遥测层仍输出 cmd 的**计算值**（便于对照），但执行器不消费
4. 通过按键 / 串口指令 / 条件编译切换模式

### 两次实验对比

| 实验 | 触发 | 预期 |
|------|------|------|
| A. 手动引导（observe 模式） | 按钮进入，人工给系统正确的物理输入 | 系统被动记录，遥测应该显示"计算控制量 ≠ 0 但不作用于物理" |
| B. 自动闭环（normal 模式） | 按钮进入，系统完整跑 | 遥测显示"计算控制量 = 0 时不作用"正常，有输出时物理应响应 |

**诊断关键**：
- A 实验的轨迹形状正确 + B 实验不正确 → **下游硬件/传感器链路 OK，问题在控制环符号**
- A 实验轨迹就不对 → 传感器/坐标变换有问题，改动目标转到 A 实验失败根因
- 两者都正确 → 症状可能是临界稳定，调参可以改善

### 对照实验设计要点

- **相同硬件 / 相同 path / 相同起始条件**：只允许"执行器通道"一个维度变化
- **遥测口径一致**：两次实验采样相同 channel，差别只在是否把 cmd 送给执行器
- **人工干预受控**：observe 模式下人给出可预期的物理输入（例如手推沿已知路径走 8 字）
- **采样分辨率**：启动瞬间（前 1~3s）采样率要高，这是最容易暴露方向错的时段

## 启动瞬间微观分析检查清单

对比两份数据（闭环 vs 观察）的前 3 秒：

1. **执行器输出方向**：闭环第一个非零控制量的符号是否符合物理期望？
2. **反馈响应方向**：执行器动后，测量值（IMU、编码器）朝哪个方向变？
3. **误差演变**：第一步后 error 是缩小还是扩大？
4. **控制量二阶变化**：第二步控制量是延续方向还是反向（反向常是过冲或符号错）？
5. **观察模式对比**：同一时刻 observe 模式下的"计算 cmd"值是多少？与闭环实际 cmd 方向一致吗？

**关键指标**：闭环启动 1s 内的 `角度变化率`、`误差变化率`、`控制量变化率` 若远超理论稳态（>3×）就是异常。

## 物理推导验证模板

**当怀疑某条通路符号错，手算一次验证**：

```
给定系统状态（起始已知）:
  measurement = 0
  desired = +A （A > 0, 小量）
  控制律计算 → output

物理上系统应往哪个方向动? 
  predict: measurement 应 +B （B > 0）

执行器收到 output 后实际应产生什么物理效果？
  actuator output 符号 → 执行器物理动作方向 → 物理响应方向

如果 predict 和 actual 方向相反 → 符号错在 ？
  - 控制律内部（sin/cos 用错象限）
  - 前馈符号宏（sign = +1/-1）
  - 执行器极性（servo 正/负对应方向）
  - 传感器符号（IMU 正/负方向定义）
```

逐项排查这 4 处，**第一个不对的点就是根因**。

## 真实案例归档

### Case 2026-04-23：平衡单车路径跟踪 lean_feedforward_sign 错误

**背景**：
- 2D 路径跟踪（Pure Pursuit / Stanley）→ 侧倾角前馈 φ → balance angle PID → servo → 车实际转向
- 原 `NAV_DEFAULT_LEAN_FEEDFORWARD_SIGN = -1.0f`，历史上某次 path 重构时 path 几何旋转 +90°，符号没跟着改

**症状**：
- 启动瞬间 yaw 错向转（段 A CCW 路径但车 CW 转）
- balance Kp=45 过大放大器把错向 ff 放大成 yaw_rate 300°/s 震荡
- 多轮调参（扩 phi_max、放开 rate_limit、加密 path 采样）都治标不治本
- DR 轨迹完全不是 8 字，车整体向 +X 方向漂

**诊断用的对照实验**：
- 新增 `manual_steer_mode` 标志 + `nav_set_manual_steer_mode()` API
- nav tracking 计算链路（PP/Stanley、DR、wp 推进、遥测）全部正常跑
- 但 `balance_set_nav_feedforward` 在 manual 模式下固定传 0
- 按钮 A（手推模式）： manual=1 + motor 正常转，用户手推沿真实 8 字走
- 按钮 B（正常模式）： manual=0 + 完整闭环跑

**对照结果**：
| 指标 | 手推（observe） | 自驱（正常 PP） | 理论 |
|------|----------------|-----------------|------|
| yaw 累计净变化 | -10.6° | +412° | 0° |
| yaw_rate 均值 | 33°/s（接近理论） | 57°/s | 29°/s |
| DR 闭合误差 | 2.16m | 5.46m | 0m |
| 轨迹形状 | 粗略 8 字 | 单向漂 | 8 字 |

→ **手推正确 + 自驱反向失败 = 闭环控制环节出错**

**物理推导**：
```
段 A CCW 起步:
  α = desired_heading - yaw = +4° (>0, desired 在 yaw 左侧)
  κ = 2·sin(α)/Ld = +0.465 (>0)
  φ_raw = arctan(v²·κ/g) = +0.82° (物理要求: 车左倾左转)
  乘 sign=-1 → φ_cmd = -0.82° (错！命令车右倾右转)
```

**修复**：`LEAN_FEEDFORWARD_SIGN` `-1 → +1`，同时降 angle PID `Kp 45 → 30`（压制下游放大器）。

**关键教训**：
1. 符号错在"historical/current"交界处最隐蔽（代码长期跑，小改动翻方向没同步）
2. 纯代码审查可能漏掉（Codex 先前 SESSION 对符号做过"PASS"结论但基于公式论证未验物理）
3. 对照实验是打破这类盲点的唯一手段
4. 启动瞬间微观数据（前 3 秒）是黄金时段，一定要看前几帧

## 速查：控制系统调试常见陷阱

| 陷阱 | 为什么错 | 正确做法 |
|------|---------|---------|
| Kp 降到很低"就能稳住"当作修复 | 符号错 + 增益压制只是掩盖症状 | 做对照实验验证符号 |
| "上一次这样改好用" | 坐标系/path 重构后符号可能需要翻转 | 从物理第一性推导重新验证 |
| 改 5 个参数一起烧录 | 效应无法分离 | 单变量实验：一次只改一个 |
| 只看稳态不看启动 | 启动瞬间最容易暴露方向错 | 遥测必须覆盖启动前 3s 高采样率数据 |
| 依赖"公式推导"审查符号 | 公式对 ≠ 物理对（坐标变换可能翻转） | 实物验证：手动给已知输入看执行器反应 |
