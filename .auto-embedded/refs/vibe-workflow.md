# Vibe 工作流补充规范

在以下场景读取本文件：

- 任务跨 2 个以上文件
- 预计需要 2 轮以上推进
- 用户要求 `多 Agent`、`主控调度`、`并行探索`、`并行验证`
- 需要把执行过程做成可回放、可回退、可交接的流程

## 1. 轮次制规则

把长任务切成多个可验收的小轮次。

- 每轮只处理一个改动点
- 开始前声明：`trace_id`、当前轮次、目标、验证标准、停止条件
- 结束时必须交付：改动摘要、涉及文件、diff 或关键变更、验证命令与输出、风险、下一轮建议
- 交接摘要压缩成 3 句话：
  1. 本轮改了什么
  2. 已确认了什么
  3. 下一轮要验证什么

默认停止红线：

- 新增依赖
- 一次改动超过 5 个文件
- 修改根目录构建/烧录/工具链配置
- 连续两轮都无法确认根因
- 需要危险操作（批量删除、强制回退、量产烧录）

## 2. 多 Agent 协议

默认分成三个角色。

### Scout

只收集证据，不写代码。

必须交付：

- 相关文件路径，按优先级排序
- 相似实现或既有模式
- 已确认约束：芯片、固件库、构建方式、引脚/时钟/DMA/中断限制
- 建议的最小改动范围

### Builder

只做实现，优先最小改动。

必须交付：

- 3-7 步执行计划
- 实际代码改动或 diff
- 实际运行过的编译/测试/烧录/日志验证结果
- 偏差说明：哪里与计划不同、为什么不同

### Verifier

只做审查和验收，不补实现。

必须交付：

- 按严重程度排序的风险清单
- 未覆盖的边界条件
- 基于 diff 和验证结果的逐项审查结论
- 最低成本补救建议

并行保护：

- 同一时刻只允许一个 Builder 写入
- Scout 与 Verifier 默认只读
- 跨角色只传：目标、约束、候选文件、证据、下一步

## 3. 证据包模板

```text
[轮次汇报]
trace_id: <任务标识>
round: <第 N 轮>
goal: <本轮目标>
verify: <成功标准>

changes:
- <改动摘要 1>
- <改动摘要 2>

evidence:
- file: <路径:行号>
- command: <实际执行命令>
- result: <关键输出或结论>

risks:
- <已知风险或未确认项>

handoff:
1. <本轮改了什么>
2. <已确认了什么>
3. <下一轮要验证什么>
```

## 4. 与四文件体系的映射

- `项目规划清单.md`：保存分轮计划、验证标准、停止条件
- `研究发现.md`：保存搜索发现、失败记录、外部资料摘要
- `编辑清单.md`：保存轮次进展、trace 和 Git 快照
- `硬件资源表.md`：保存芯片、引脚、DMA、中断、时钟等稳定事实

建议写法：

```text
trace_id=<id> | round=<n> | role=<Scout/Builder/Verifier> | action=<摘要> | verify=<命令/标准> | result=<pass/fail/pending>
```

## 5. 证据优先原则

- 没有代码位置、日志、构建输出、测试结果、数据手册或网表依据时，不下最终结论
- 没有验证结果时，不说“应该没问题”
- Verifier 只能基于 diff、实际执行结果、覆盖情况和已知约束给结论
