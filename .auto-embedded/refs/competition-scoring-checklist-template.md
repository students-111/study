# 竞赛评分点 → 验收表通用模板

> **元层文档**：任何竞赛题的"评分细则"都能套这个模板生成"100 分逐项打钩"验收表。
>
> 与 `.auto-embedded/refs/competition-task-router.md` 配合用：路由器判定题型 → 本文档对应章节生成验收表。
>
> 设计原则：评分点 = acceptance test。每个评分点必须能用"做了 / 没做"二元判定，禁止"差不多""基本上"。

---

## 0. 通用使用流程

```
输入：题目评分细则原文（命题书的"评分细则"章节）
   ↓
Step 1：把每条评分点拆成"操作 → 验证位置 → 预期 → 设计依据 → 分值"5 元组（v2 升级）
Step 2：用 §1 通用模板填空生成表格
Step 3：按题型用 §2-§7 的专项增补（A-F 6 题型）
Step 4：放进 **docs/checklist-100分.md**（不是项目规划清单顶部 — v2.1 修订），[QA] Agent CP-3 阶段逐项打钩
   ↓
输出：100 分逐项追踪表（打印纸质版，赛前 24h 最后一道闸）
```

> **位置约定**（v2.1）：100 分验收表独立到 `docs/checklist-100分.md`。`项目规划清单.md` 顶部留给 `competition_state` YAML 块（见 `.auto-embedded/refs/contracts.md §比赛状态机`），不再混存验收表。

## 1. 通用 5 元组模板 ★v2 升级（加 why-evidence 字段）

每条评分点统一格式：

```markdown
| 评分点 | 操作 | 验证位置 | 预期输出 / 条件 | 设计依据 | 分值 | ✓/✗ |
|---|---|---|---|---|---|---|
```

**5 元组字段定义**：

| 字段 | 含义 | 反例 |
|---|---|---|
| **操作** | 具体可执行命令 / 按键 / 输入 | ❌ "正常运行" → ✅ "输入 start 命令" |
| **验证位置** | 在哪里看结果（串口 / OLED / TF 卡 / 示波器 / LED） | ❌ "效果" → ✅ "示波器 CH1" |
| **预期输出** | 字面值 / 范围 / 字符串 | ❌ "正确" → ✅ "误差 ≤ ±1%" 或 "返回 RTC Config success" |
| **设计依据** ★新 | 为什么这样设计 / 哪份证据支持 | ❌ "AI 说要这样" → ✅ "见 编辑清单_MATLAB.md 第 3 轮仿真 + 实测对比图" |
| **分值** | 该项分数（必须有，反推优先级）| ❌ "重要" → ✅ "5 分" |

### 1.1 为什么加"设计依据"字段（关键 — 防答辩翻车）

答辩老师最常问的不是"做了什么"而是"**为什么这样做**"。"设计依据"列必须填**可追溯到磁盘记忆 / 仿真曲线 / 实测数据**的指针。**禁止填"经验"/"参考资料"/"AI 推荐"**。

**格式约定（v2.1 强制）**：用 `file:line` 形式（如 `scripts/lqr_design.m:28`、`编辑清单_MATLAB.md:42-58`），不要用自然语言"第 X 行" — 让 QA 可机器校验路径存在性 + 行号合法性。

| 评分点 | ✅ 合格"设计依据" | ❌ 不合格 |
|---|---|---|
| LQR 控制器 | `scripts/lqr_design.m:28-45` Q/R 选取 + `figures/lqr_poles.png` 极点位置 | "AI 给的参数" |
| 50 Hz 低通滤波 | `编辑清单_MATLAB.md:42-58` FFT 实测主噪声 47 Hz | "电源工频是 50 Hz" |
| 用 Hough 圆检测 | `算法报告.md:§3.2` HSV vs Hough 对比，召回率 95% vs 67%（`log/vision_eval_20260520.csv`）| "Hough 更准" |
| Q15 定点不用 float | `硬件资源表.md:#L12` MCU 型号 STM32F103 无 FPU | "嵌入式用定点" |

**QA 二次校验**：CP-3 阶段 QA 必须对每条"设计依据" `file:line` 跑 `test -f "$file" && wc -l "$file"`（或等价检查），若路径不存在或行号超界 → 打 Defect Ticket，severity=high。

### 1.2 通用模板填空示例（5 元组完整版）

题目原文（如）："1.3 系统串口输入 test 后，应在 1 秒内返回完整自检结果，含 flash / TF 卡 / RTC 三项状态"

转换成 **5 元组**（操作 / 验证位置 / 预期 / **设计依据** / 分值）：

```markdown
| 评分点 | 操作 | 验证位置 | 预期输出 / 条件 | 设计依据 | 分值 | ✓/✗ |
|---|---|---|---|---|---|---|
| 1.3 test 命令自检 | 串口输入 `test` | 串口（115200）| 1s 内返回 `flash...ok` + `TF card...ok/error` + `RTC: yyyy-mm-dd hh:mm:ss` | 见 编辑清单_DRV.md（drv_flash 自检 + drv_sd 探测 + drv_rtc 时钟） + 实测 log/test_cmd_20260520.txt | 4 | ☐ |
```

**禁止**：填空示例只填 4 列省略"设计依据"（早期 v1 习惯，v2 已禁）。每条 5 元组都必须有指向磁盘记忆 / 仿真 / 实测的指针。

## 7.5 答辩主观分专项表 ★新（why-evidence chain）

5 元组只能管功能演示分（约 60-80%）。剩余答辩 + 技术方案分（20-40%）必须用本专项表。

### 7.5.1 答辩 10 个 why 表（每题必填）

每个 why 必须绑定 2 类证据：**仿真证据**（MATLAB / 仿真曲线）+ **实测证据**（示波器照片 / 串口日志 / 视频）。**禁止只靠"AI 说"**。

| # | 问题 | 回答模板（≤ 60 字）| 仿真证据 | 实测证据 | 备用回答（追问） |
|---|---|---|---|---|---|
| 1 | 为什么用 X 算法 / 拓扑 / 方案？ | 例：选 LQR 因为 4 状态多输入耦合 | scripts/X_design.m | 实测响应 vs 仿真曲线 | 如果 PID 也行，为何不用？性能差 N% |
| 2 | 为什么采样率 / 控制频率取 Y？ | 例：闭环带宽 5 Hz，10×→ 50 Hz | bode 图 | 示波器测控制周期 | Nyquist 给的下限是 X，10× 是工程经验 |
| 3 | 为什么用 Z 滤波器 / 阶次？ | 例：Butterworth 4 阶 + fc=20 Hz | freqz 曲线 | 滤波前后 RMS 对比 | 高阶相位畸变，低阶衰减不够 |
| 4 | 为什么 Q/R 取这组值？ | 例：Bryson 法则起点 + 3 轮实测调 | scripts/lqr_design.m | 调参曲线 | Q 各项物理意义 |
| 5 | 为什么 ADC 用 12-bit 而非 16-bit？ | 例：内置 + DMA 够 | datasheet 引用 | ADC SNR 实测 | 16-bit 外置需要 SPI + 板面 |
| 6 | 抗干扰怎么做？ | 例：硬件 RC + 软件中值 | scipy/MATLAB 滤波仿真 | 加扰前后 RMS | 中值 vs 均值差异 |
| 7 | 实时性怎么保证？ | 例：50 Hz 中断 + 任务 <10ms | scope CPU 占用 | 实测中断周期 | 最坏情况栈深度 |
| 8 | 失效安全 / 看门狗策略？ | 例：IWDG 500ms + ERROR 状态 | 状态机图 | 故意触发 fault 录像 | 看门狗周期为何 500ms |
| 9 | 代码可维护性？ | 例：分层 L1-L6 + arch-check 通过 | include 依赖图 | arch-check.sh 输出 | 为何不全堆 main.c |
| 10 | 创新点 / 加分项？ | 例：自校准 / 多模式 / GUI 上位机 | 演示视频 | 现场演示 | 与同类方案比较 |

### 7.5.2 技术方案文档要点（25% 分数）

| 必含章节 | why-evidence 关键 | 谁写 |
|---|---|---|
| 题目分析 + 指标 | 量化每个指标，引用题目原文 | [REPORT] |
| 系统方案框图 | 引用接口契约 + 硬件资源表 | [ARCH] |
| 关键算法原理 | 公式 + 仿真曲线 + 算法选型对比 | [MATLAB] |
| 硬件方案 | BOM + 选型理由（含对比的替代品）| [DRV] |
| 测试结果 | 5 元组验收表打钩状态 + 误差曲线 | [QA] |
| 创新点 | 用对比数据说明，禁"我们做了 X" | [REPORT] |

### 7.5.3 答辩演练强制流程（[REPORT] CP-5 必做）

```
Step 1：列出本题的所有"为什么 X"问题（≥ 10 个）
Step 2：每个 why 填上"仿真证据 + 实测证据"路径
Step 3：模拟答辩 3 轮：
  Round 1: 自己讲一遍录像
  Round 2: 队友扮老师追问
  Round 3: 全队跑一次"无 AI 回答"演练（禁止说"AI 说"）
Step 4：每个 why 通过率 ≥ 80% 才能上场
```

## 8.5 ⭐ 临场变更单（Delta Form）★新

赛场常见：评测员临场加题 / 减项 / 改测试条件。如果验收表是固定不可变的，每次改动都要全部重做。**v2 加变更单流程**让 delta 可追溯。

### 8.5.1 变更单模板

```markdown
# 变更单 #N（赛场更新）
时间: 2026-08-03 14:30
评测员说: "<原话引用>"

# 影响分析
受影响评分点:
  - 删除: [#A.3, #B.5]                # 引用 5 元组 ID
  - 新增: [#NEW-1]                    # 评测员临场加的项
  - 修改: [#C.2 预期值改为 ±0.5%]     # 已有项条件变更
  - 重测: [#D.1, #D.4]                # 改前已 ✓，需要重新验证

# 受影响 Agent
  - [ALG]: 需要改 svc_xxx.c 适应新条件
  - [QA]: 重测 D.1 / D.4
  - [REPORT]: 报告对应章节改

# 分数 delta
变更前: __ / 100
变更后: __ / 100
净影响: ± __ 分

# 修复时间预算
预计耗时: __ 分钟 / 小时
余下竞赛时间: __ 小时

# 决策
[ ] 做（值得）
[ ] 不做（性价比低，放弃 X 分）
[ ] 部分做（只改 [#NEW-1]，舍 [#A.3]）

# Git 处理
重测项 → git tag v0.3-fix<N> 标注重测点
新增/修改 → 进新一轮 [ALG] [QA] 修复
```

### 8.5.2 变更单触发条件

| 触发 | 流程 |
|---|---|
| 评测员现场加题 | 加变更单 + 派 [ALG] 微修复 + [QA] 重测 |
| 评测员现场减项 | 加变更单 + 删 5 元组项 + 重算总分 |
| 测试条件变（电压 5V→3V）| 加变更单 + 评估硬件 / 软件影响范围 |
| 队伍发现 bug 改条件 | 不加变更单（这是 [QA] FAIL→修复轮，走原流程）|

### 8.5.3 变更单存档位置

```
docs/
```

### 8.5.4 决策门：变更单要不要做

```
评测员临场加题
   ↓
[ARCH] 算 delta（5 元组改动量 + Agent 影响 + 时间预算）
   ↓
分支：
  ├── 时间够 + 加分多 → 做，进修复轮
  ├── 时间紧 + 加分少 → 放弃，明确告诉评测员"本项不展示"
  └── 时间紧 + 加分多 → 让 [QA] 决定能否最小修复（只改关键点）
```

## 9. 评分点提取技巧（应对题目模糊）

### 9.1 题目"评分细则"章节直接抄

最简单：题目命题书的"评分细则"或"评测流程"章节，逐条抄过来变 4 元组。

### 9.2 题目正文里的隐含指标

很多题目正文里说"应该 / 必须 / 不得"等关键词，是隐含评分点：

| 正文关键词 | 评分点 |
|---|---|
| "必须支持 ..." | 1 个功能点 |
| "应在 X 秒内 ..." | 1 个时间指标 |
| "精度 ± X" | 1 个精度指标 |
| "不得超过 ..." | 1 个边界检查 |
| "应能恢复 ..." | 1 个鲁棒性测试 |

### 9.3 题目示例图 / 表格

题目里的"示例输出图"对应一条评分点（输出格式必须与样图一致）。

### 9.4 用 [QA] Agent 把所有评分点抽出

```
[QA] 启动 prompt：
"输入：题目原文 + 评分细则
任务：列出所有评分点，每条按 4 元组（操作/位置/预期/分值）输出。
检查覆盖度：分值合计必须等于满分 100 分（如不等于报告差额）"
```

## 11. 失败兜底

| 问题 | 处置 |
|---|---|
| 评分细则太模糊（"完成度高得高分"）| 找命题书细则段 / 看历年同类题对照 |
| 分值合计 ≠ 100 | 漏写或题目设计有溢出，标记并向评测员澄清 |
| 主观分（创新点 / 答辩）怎么打钩 | 用"准备了 / 没准备"+ 答辩演练通过率代替 |
| 评测员临场加题 | 留 1 个空白行预案 |
| 评测员减项 | 主动确认"哪几项今天不演"避免空跑 |
