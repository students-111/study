# 系统化调试方法论（四阶段根因分析）

> 本文件在 EXECUTE 阶段遇到问题、三次失败协议触发回退 RESEARCH 时按需加载。
> 与 `.auto-embedded/refs/troubleshooting.md`（症状速查表）互补使用：速查表解决常见问题，本文件处理复杂/未知问题。

---

## 铁律

**不找到根因，不尝试修复。** 症状修复是失败。

---

## Phase 1: 根因调查

**目标**：收集证据，缩小问题范围

1. **完整阅读错误信息**
   - 编译错误：从第一个 error 开始读（后续错误可能是连锁反应）
   - 运行时异常：读取 HardFault/MemManage/BusFault 的寄存器转储
   - 硬件异常：记录具体现象（无输出/乱码/电平异常/时序偏差）

2. **一致复现**
   - 能否 100% 复现？还是间歇性？
   - 间歇性问题 → 重点排查：竞态条件、中断抢占、电源毛刺、时钟抖动
   - 记录复现条件到 `研究发现.md`

3. **检查近期变更**
   - `git diff HEAD~3` 查看最近改动
   - 是否在改动之前就存在问题？用 `git stash` / `git checkout` 二分定位

4. **收集硬件证据**（嵌入式特有）
   - 示波器：测量关键信号波形（时钟、通信总线、PWM）
   - 逻辑分析仪：抓取 I2C/SPI/UART 协议数据
   - 串口日志：在关键路径添加诊断输出
   - 万用表：确认供电电压、参考电压

5. **追踪数据流**
   - 从输入到输出，逐步验证每个环节
   - 时钟链：HSE/HSI → PLL → SYSCLK → AHB → APBx → 外设
   - 数据链：传感器 → ADC → DMA → 缓冲区 → 算法 → DAC/PWM

---

## Phase 2: 模式分析

**目标**：对比工作与不工作的代码，定位差异

1. **找到工作的参考示例**
   - 芯片厂商官方例程（最权威）
   - 已验证的项目代码（硬件资源表中记录的已完成模块）
   - grok-search 搜索同型号芯片的工作示例

2. **逐行对比差异**
   - 寄存器配置：逐位对比（CR、SR、DR 等）
   - 初始化顺序：某些外设对初始化顺序敏感（如 DMA 必须在外设使能前配置）
   - 时序要求：建立/保持时间、最小脉宽、去抖延时

3. **理解依赖链**
   - 时钟依赖：外设时钟是否已使能？分频系数是否正确？
   - GPIO 依赖：复用功能是否正确配置？是否存在引脚冲突？
   - 中断依赖：NVIC 是否使能？优先级是否合理？中断向量是否正确？

4. **记录对比结果**到 `研究发现.md`

---

## Phase 3: 假设验证

**目标**：形成单一假设并用最小实验验证

1. **形成单一假设**
   - 基于 Phase 1-2 的证据，提出最可能的根因
   - 假设必须具体可测试，如"USART1 时钟分频错误导致波特率偏差"

2. **设计最小化测试**
   - 隔离变量：一次只改一个配置
   - 控制对照：保留工作代码作为对比
   - 嵌入式常用测试手段：
     - GPIO 翻转 + 示波器测量执行时间
     - 串口打印关键寄存器值
     - 断点调试查看变量值
     - LED 闪烁模式指示执行路径

3. **验证假设**
   - 假设成立 → 进入 Phase 4 修复
   - 假设被证伪 → 记录排除原因到 `研究发现.md`，返回 Phase 1 或提出新假设
   - **禁止同时测试多个假设**（无法判断哪个修复生效）

---

## Phase 4: 修复实施

**目标**：基于已验证的根因，实施最小修复

1. **单一修复原则**
   - 一次只修改一处
   - 修改前记录当前状态（Git 快照或手动记录寄存器值）

2. **验证修复**
   - 重新运行 Phase 1 的复现测试
   - 确认问题消失
   - 确认没有引入新问题（回归测试）

3. **修复失败判定**
   - 如果 Phase 4 修复 3 次仍失败 → **质疑架构**
   - 可能需要更换实现方式（轮询→中断、软件→DMA、bitbang→硬件外设）
   - 此时触发三次失败协议，写入 `研究发现.md` 后回 PLAN 重新规划

---

## 速查：调试常见陷阱

| 陷阱 | 正确做法 |
|------|---------|
| 同时改多处然后测试 | 一次只改一处 |
| 跳过复现直接猜原因 | 先 100% 复现再分析 |
| 忽略间歇性故障 | 间歇性 = 竞态/时序/电源问题，更需要系统排查 |
| "上次这样改好用" | 每次环境不同，必须从证据出发 |
| 只看软件不看硬件 | 软硬件同时排查，示波器是你的朋友 |
| 编译通过就认为修好了 | 编译通过 ≠ 功能正确 |

---

## 与其他机制的关联

- **三次失败协议**：Phase 4 连续失败 3 次时触发，停止重试并回退 RESEARCH
- **研究发现.md**：每个 Phase 的关键发现必须写入，防止循环调试时遗忘线索
- **troubleshooting.md**：常见问题先查速查表，速查表无法解决的问题再用四阶段方法
- **硬件资源表.md**：Phase 2 对比时的权威参考，修复后如有硬件变更需同步更新
- **control-loop-sign-debug.md**：闭环控制系统（PID/前馈）特化调试。当症状是"调参调不好 + 方向反 + 震荡"组合时按该文件的"观察模式对照实验法"排查，避免陷入盲目调参
