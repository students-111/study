# 网表读取模式

> 触发词：`网表` / `netlist` / `读网表` / `查网表`
>
> 用途：从 EDA 工具导出的网表文件中提取 MCU 引脚连接关系，自动生成引脚映射表，与硬件资源表比对，确保代码中的引脚配置与实际原理图一致。
>
> **核心原则**：有网表就读网表，别手动猜引脚；没网表就问用户要，实在没有再查手册。

---

## 适用场景

- RESEARCH 阶段进行引脚规划前，自动检测并读取网表
- EXECUTE 阶段编写引脚配置代码时，交叉验证网表与代码一致性
- 用户主动触发：`网表` / `netlist` / `读网表` / `查网表`

## 支持的网表格式

### 官方 EDA 导出格式

| EDA 工具 | 导出格式 | 文件扩展名 | 格式特征 |
|---------|---------|-----------|---------|
| **嘉立创EDA（标准版）** | Protel/Altium 格式 | `.net` | `[`/`]` 包围元件，`(`/`)` 包围网络，`元件-引脚号` 连接 |
| **嘉立创EDA（标准版）** | Pads 格式 | `.asc` | Pads ASCII 格式 |
| **嘉立创EDA（标准版）** | FreePCB 格式 | `.net` | FreePCB 文本格式 |
| **嘉立创EDA（标准版）** | Spice 仿真格式 | `.cir`/`.spice` | SPICE 网表 |
| **嘉立创EDA（专业版）** | 嘉立创EDA 原生格式 | `.net` | 专业版原生格式 |
| **嘉立创EDA（专业版）** | Allegro 格式 | `.dat` | Cadence Allegro 格式 |
| **嘉立创EDA（专业版）** | PADS 9.5 格式 | `.asc` | Mentor PADS 格式 |
| **Altium Designer** | Protel 格式 | `.NET`/`.NetList` | 同嘉立创标准版 Protel 格式 |
| **KiCad** | S-Expression 格式 | `.net`/`.kicad_net` | `(export` 开头的 S-Expression |

> **优先级**：Protel/Altium 格式（嘉立创最常用导出格式）> KiCad > 其他。遇到不认识的格式时，输出文件前 50 行请用户确认格式。
>
> **格式来源**：[嘉立创EDA标准版-导出网表](https://docs.lceda.cn/cn/Schematic/Export-NetList/index.html) | [嘉立创EDA专业版-导出网表](https://prodocs.lceda.cn/cn/schematic/export-netlist/)

## 阶段一：检测网表文件

### 1.1 自动搜索

在工程根目录递归搜索网表文件，使用以下 Glob 模式：

```
**/*.net
**/*.NET
**/*.NetList
**/*.kicad_net
**/*.asc
**/*.cir
**/*.spice
**/*.dat
**/Netlist*.*
**/*netlist*.*
**/*网表*.*
```

同时检查常见 EDA 输出目录：
```
hardware/
pcb/
schematic/
eda/
docs/hardware/
```

> **过滤规则**：排除 `node_modules/`、`.git/`、`build/`、`output/` 等非 EDA 目录。对 `.dat` 文件需检查内容是否为 Allegro 网表格式，避免误匹配。

### 1.2 判断与降级

```
找到网表文件
├── 找到 1 个 → 直接进入阶段二
├── 找到多个 → 列出所有文件，询问用户选择哪个（优先推荐最近修改的）
└── 未找到
    ├── 询问用户："工程目录中未检测到网表文件。你是否有网表文件？请提供路径。"
    ├── 用户提供路径 → 验证文件存在后进入阶段二
    └── 用户无法提供 → 降级处理（见降级策略）
```

### 1.3 降级策略（无网表可用时）

当用户无法提供网表时，按以下顺序获取引脚信息（与主协议容灾矩阵一致）：

1. **grok-search**：搜索官方数据手册中的引脚定义
   ```
   先 resolve-library-id 找到芯片库 ID
   再 query-docs 查询 "pin definition alternate function mapping"
   ```

2. **Document Skills** (`/pdf`)：提取本地数据手册中的引脚映射表
   ```
   搜索 "<芯片型号> pin mapping GPIO alternate functions"
   ```

3. **Sequential Thinking MCP**（grok-search 不可用时）：结构化推理引脚分配
   ```
   调用 sequentialthinking 分析芯片引脚复用功能，推理推荐分配方案
   ```

4. **手动规划**（以上均不可用时）：回退到 `.auto-embedded/refs/pin-planning.md` 的标准引脚规划流程

**降级执行要求**（与主协议一致）：
- **记录降级事件**：在 `编辑清单.md` 中记录："无网表文件，引脚信息来源: <grok-search/Document Skills/Sequential Thinking/手动规划>，降级原因: <具体原因>"
- **验证结果质量**：使用备用工具后必须评估搜索结果是否满足需求
- **服务恢复监控**：后续阶段定期检测主工具可用性，恢复后优先使用主工具

## 阶段二：解析网表

### 2.1 格式识别

读取文件前 20 行，自动判断格式：

| 特征 | 判定格式 |
|------|---------|
| 前几行含 `[` 开头的元件块，后续含 `(` 开头的网络块 | **Protel/Altium 格式**（嘉立创标准版/AD 最常用） |
| 含 `(export` 或 `(net` S-Expression 语法 | **KiCad 格式** |
| 含 `*PART*` 和 `*NET*` 段标记 | **Pads 格式** |
| 含 `.SUBCKT` 或以 `*` 开头的注释行 | **Spice 格式**（通常不用于引脚提取，提示用户选其他格式） |
| 无法识别 | **兜底**：输出前 50 行，询问用户确认 |

### 2.2 Protel/Altium 格式解析（嘉立创最常用）

Protel 格式网表是纯文本 ASCII 文件，结构分两部分：

**元件定义部分**（`[` 和 `]` 包围）：
```
[
U1
LQFP48
]
[
R1
RES0603
]
[
C1
CAP0402
]
```
- 第 1 行：元件位号（Designator），如 `U1`、`R1`
- 第 2 行：封装名（Footprint），如 `LQFP48`、`RES0603`

**网络连接部分**（`(` 和 `)` 包围）：
```
(
USART1_TX
  U1-30
  J1-1
)
(
I2C1_SCL
  U1-42
  U2-6
)
(
GND
  R1-2
  C1-2
  U1-23
)
```
- 第 1 行：网络名（Net Name），如 `USART1_TX`
- 后续行：`元件位号-引脚编号` 格式的连接列表

**解析逻辑**：
1. 解析 `[`/`]` 块，建立 `元件位号 → 封装名` 映射
2. 解析 `(`/`)` 块，按网络名分组，建立 `网络名 → [(元件位号, 引脚编号)]` 映射
3. 忽略电源网络（VCC/VDD/GND/VSS/3V3/5V）的详细分析，仅记录

> **注意**：Protel 格式网表中只有引脚编号，没有引脚名（如 PA9）。需要结合芯片封装的引脚编号到引脚名的映射表（从数据手册获取（grok-search 搜索 + Document Skills 提取））才能得到引脚名。

### 2.3 KiCad 格式解析

KiCad 网表使用 S-Expression 格式：
```
(export (version D)
  (components
    (comp (ref U1) (value STM32F103C8T6) (footprint LQFP48))
    (comp (ref R1) (value 10k) (footprint RES0603))
  )
  (nets
    (net (code 1) (name "USART1_TX")
      (node (ref U1) (pin 30))
      (node (ref J1) (pin 1))
    )
  )
)
```

**解析逻辑**：
1. 解析 `components` 节点，建立元件索引（位号、值、封装）
2. 解析 `nets` 节点，建立网络连接关系
3. 通过 `comp` 中的 `value` 字段识别 MCU

### 2.4 通用兜底解析

对于无法自动识别的格式：
1. 输出文件前 50 行
2. 询问用户确认格式和关键字段含义
3. 根据用户描述进行解析

## 阶段三：提取 MCU 引脚映射

### 3.1 识别 MCU 元件

从网表的元件列表中，通过以下特征定位 MCU：

| 识别方式 | 匹配规则 |
|---------|---------|
| 元件值（Value）（KiCad 格式可用） | 匹配 `STM32*`、`ESP32*`、`CH32*`、`GD32*`、`AT32*`、`ATmega*`、`MSP430*`、`nRF*`、`RP2040` 等 |
| 封装（Footprint） | LQFP、QFP、QFN、BGA 等大引脚数封装（Protel 格式主要靠此判断） |
| 元件位号（Ref） | 通常为 `U1`、`U2` 等，需结合封装判断 |

> **Protel 格式特殊处理**：Protel 格式只有位号和封装，没有元件值。通过封装（如 `LQFP48`、`LQFP64`、`LQFP100`）识别 MCU，必要时询问用户确认哪个是 MCU。

找到多个可能的 MCU 时，列出候选并询问用户确认。

### 3.2 引脚编号到引脚名映射

**Protel 格式的关键步骤**：网表中只有引脚编号（如 `30`），需要将编号转换为引脚名（如 `PA9`）。

获取映射表的优先级：
1. **项目中已有的数据手册 PDF**：用 /pdf skill 读取引脚定义表
2. **grok-search**：搜索 `"<芯片型号> <封装> pin number to pin name mapping"`
3. **grok-search**：搜索 `"<芯片型号> <封装> pinout pin assignment"`
4. **用户提供**：询问用户是否有引脚映射表

> **KiCad 格式**：通常也只有引脚编号，同样需要此步骤。但如果 KiCad 网表中 `pin` 字段直接是引脚名（如 `PA9`），则可跳过。

### 3.3 生成引脚映射表

提取 MCU 的所有引脚连接，生成结构化映射表：

```
=== MCU 引脚映射表（从网表提取） ===
来源: <网表文件名>
MCU: <元件位号> (<芯片型号/封装>)
提取日期: <日期>

| 引脚编号 | 引脚名 | 网络名 | 连接到 | 功能推断 |
|---------|--------|--------|--------|---------|
| 30      | PA9    | USART1_TX | J1-1 | USART1 发送 |
| 31      | PA10   | USART1_RX | J1-2 | USART1 接收 |
| 42      | PB6    | I2C1_SCL  | U2-6 | I2C1 时钟 |
| 43      | PB7    | I2C1_SDA  | U2-7 | I2C1 数据 |
| 10      | PA0    | BAT_ADC   | R3-1 | ADC 电压采样 |
| 48      | VDD    | VCC       | 电源 | - |
| 23      | VSS    | GND       | 地 | - |
```

### 3.4 功能推断规则

根据网络名和对端元件自动推断引脚功能：

| 网络名特征 | 推断功能 |
|-----------|---------|
| 含 `UART`/`USART`/`TX`/`RX` | 串口通信 |
| 含 `I2C`/`SCL`/`SDA` | I2C 通信 |
| 含 `SPI`/`MOSI`/`MISO`/`SCK`/`CS` | SPI 通信 |
| 含 `PWM`/`MOTOR`/`MOT` | PWM 输出/电机控制 |
| 含 `ADC`/`AIN`/`ANALOG`/`BAT` | ADC 采样 |
| 含 `LED`/`KEY`/`BTN`/`BUTTON` | GPIO（LED/按键） |
| 含 `ENC`/`ENCODER` | 编码器输入 |
| 含 `CAN` | CAN 总线 |
| 含 `USB`/`DP`/`DM` | USB 通信 |

> 推断结果仅供参考，标注 `[推断]` 标签。涉及复用功能冲突时仍需查数据手册确认。

## 阶段四：比对硬件资源表

### 4.1 有硬件资源表时

读取 `硬件资源表.md`，逐项对比：

```
比对结果：
一致 PA9 → USART1_TX：网表与资源表一致
一致 PB6 → I2C1_SCL：网表与资源表一致
冲突 PA0 → 网表为 BAT_ADC（ADC采样），资源表为 TIM2_CH1（PWM）→ 冲突！
遗漏 PB3 → 网表有连接（SPI1_SCK），但资源表中未记录 → 遗漏！
```

- **一致**：无需操作
- **冲突**：高亮警告，提示用户确认以哪个为准（网表反映实际硬件，通常以网表为准）
- **遗漏**：提示补充到资源表

### 4.2 无硬件资源表时

直接用网表数据生成 `硬件资源表.md` 的引脚分配表部分，格式参考 `.auto-embedded/refs/checklist-templates.md`。

## 阶段五：应用到代码

### 5.1 生成引脚定义宏

基于网表提取结果，生成 C 语言引脚定义：

```c
/**
 * @brief 引脚定义 - 来源: 网表文件 <文件名>
 * @note  自动从网表提取，与原理图一致
 */

/* 串口引脚 */
// @netlist <文件名>, 网络名: USART1_TX, 对端: J1-1（调试串口连接器）
#define USART1_TX_PIN       GPIO_Pin_9
#define USART1_TX_PORT      GPIOA
// @netlist <文件名>, 网络名: USART1_RX, 对端: J1-2
#define USART1_RX_PIN       GPIO_Pin_10
#define USART1_RX_PORT      GPIOA

/* I2C 引脚 */
// @netlist <文件名>, 网络名: I2C1_SCL, 对端: U2-6（MPU6050 SCL）
#define I2C1_SCL_PIN        GPIO_Pin_6
#define I2C1_SCL_PORT       GPIOB
// @netlist <文件名>, 网络名: I2C1_SDA, 对端: U2-7（MPU6050 SDA）
#define I2C1_SDA_PIN        GPIO_Pin_7
#define I2C1_SDA_PORT       GPIOB
```

### 5.2 注释规范

所有从网表提取的引脚配置，注释中必须标注来源：

```c
// @netlist <网表文件名>, 网络名: <NET_NAME>, 对端: <对端元件-引脚编号>
```

### 5.3 代码验证

在 EXECUTE 阶段写完引脚配置代码后，自动比对：
- 代码中的引脚定义 vs 网表中的引脚映射
- 确保无遗漏、无错配
- **若发现不一致**：
  - 仅实现错误（引脚写错）→ 留在 EXECUTE 修正，展示审查门后继续
  - 涉及硬件方案变更（网表与需求不匹配）→ 回 PLAN 重新规划
- 自动比对应在"展示变更给用户"之前完成

## 自动触发规则

### 在 RESEARCH 阶段步骤 7（引脚规划）前自动触发

当 RESEARCH 阶段进行到引脚规划步骤时，自动执行阶段一（检测网表文件）：

```
if (工程目录存在网表文件) {
    执行网表读取模式（阶段一→四）
    将提取的引脚映射直接填入硬件资源表的引脚分配表
    // 跳过"手动收集引脚连接信息"，但不跳过后续约束验证
    继续执行 SKILL.md 步骤 7 的后半段：
      - 检查调试口（SWDIO/SWCLK）和烧录口是否被占用
      - 检查 STRAP 引脚是否正确配置
      - 检测引脚冲突（同一引脚的多重复用功能）
      - 生成冲突警告（如有）
} else {
    询问用户是否有网表
    if (用户提供网表) {
        执行网表读取模式
        继续约束验证
    } else {
        降级到标准引脚规划流程（.auto-embedded/refs/pin-planning.md）
        用 grok-search → Document Skills → Sequential Thinking 查询芯片引脚信息
    }
}
```

> **关键区分**：网表模式替代的是"收集引脚连接信息"这一步，不替代"约束验证与冲突检测"。SWD/STRAP/复用冲突检查始终执行。

### 在 EXECUTE 阶段编写引脚配置代码时

自动检查是否已有网表提取的引脚映射数据：
- 有 → 引用映射数据生成引脚定义宏，比对完成后再展示审查门
- 无 → 按常规流程手动配置

## 核心规则

| # | 规则 | 说明 |
|---|------|------|
| 1 | **网表优先** | 有网表时以网表为引脚配置的唯一真相源，网表反映实际硬件连线 |
| 2 | **格式自适应** | 优先 Protel 格式（嘉立创最常用），不认识的格式输出前 50 行让用户确认 |
| 3 | **MCU 自动识别** | 从封装/元件值自动定位 MCU，多个候选时询问用户 |
| 4 | **引脚编号需映射** | Protel 格式只有引脚编号，必须查数据手册获取引脚名后才能用于代码 |
| 5 | **功能推断标注** | 自动推断的功能标注 `[推断]`，关键配置需用户确认 |
| 6 | **约束验证不跳过** | 即使有网表，SWD/STRAP/复用冲突检查仍必须执行 |
| 7 | **比对同步** | 网表数据必须与硬件资源表同步，发现不一致立即警告 |
| 8 | **来源溯源** | 代码中从网表得到的引脚配置必须用 `@netlist` 注释标注来源 |
| 9 | **降级记录** | 无网表时的降级方案必须记录在编辑清单中，降级链路与主协议一致 |
